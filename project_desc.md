# Sonar: A Terminal-Native Distributed Trace Visualizer

## 1. Project Overview

Sonar (working title) is a lightweight, terminal-based User Interface (TUI) designed to visualize distributed traces for local microservices development. It acts as a local OpenTelemetry (OTel) collector, catching telemetry data emitted by backend services and rendering the execution path of a single request across multiple network boundaries (HTTP, gRPC, and message brokers) in real-time, entirely within the command line.

## 2. The Problem Statement (What it Solves)

When developing microservices locally, tracking the lifecycle of a single request is incredibly difficult. For example, if an API Gateway receives a request, makes a synchronous gRPC call to a secondary service, and then publishes an asynchronous event to a Kafka topic, observing that entire flow requires significant overhead.

### Current Pain Points:

- **The Log Scavenger Hunt:** Developers are forced to keep multiple terminal tabs open, tailing the raw logs of different services and manually copy-pasting correlation IDs to stitch the narrative together.

- **Heavy Observability Tooling:** The standard solution is to run cloud-native tools like Jaeger or Zipkin locally. These require spinning up heavy Docker containers, relying on a web browser, and navigating complex UIs just to check a simple local test request.

- **Context Switching:** Moving away from the terminal (where the code is run and built) to a browser breaks the developer's focus and flow.

## 3. The Solution (How it Solves It)

Sonar bridges the gap between raw terminal logs and heavy web dashboards. It runs directly in the terminal alongside the developer's services.

By running an embedded OpenTelemetry receiver, it intercepts the standardized trace spans generated by the services. It instantly processes this data in memory and draws an interactive, hierarchical ASCII tree representing the request's journey. It isolates errors and reveals exact network payloads with zero context-switching.

| Feature | Raw Terminal Logs (`tail -f`) | Web UIs (Jaeger/Zipkin) | Sonar (This Project) |
|---------|-------------------------------|-------------------------|--------------------------|
| Setup Overhead | None | High (Requires Docker/Images) | Low (Single binary/script) |
| Trace Stitching | Manual (Mental mapping) | Automatic | Automatic |
| Interface | Text output | Heavy Web Browser | Fast Terminal UI |
| Resource Usage | Low | High | Low |

## 4. Core Features

- **OTLP HTTP/gRPC Ingestion:** Runs a background server that natively accepts OpenTelemetry Protocol (OTLP) data, requiring zero custom configuration on the backend services other than pointing the standard OTel agent to localhost.

- **Live ASCII Dependency Tree:** Visually maps the parent-child relationships of spans (e.g., Gateway └── Billing Service └── Kafka Topic) using Unicode box-drawing characters.

- **Instant Payload Inspection:** A dedicated side-panel that displays the raw metadata associated with a selected node, including HTTP headers, gRPC statuses, database query strings, and Kafka partition details.

- **Error State Bubbling:** Automatically highlights failed spans (e.g., in red or with an [X] indicator) and propagates that visual warning up the tree so developers can instantly spot where a distributed request died.

- **Fuzzy Search & Filtering:** Quickly search through recent traces by HTTP path, service name, or specific `trace_id`.

## 5. Architecture & Working Mechanism (Under the Hood)

The system is decoupled into two primary domains: the telemetry generation (the microservices) and the telemetry visualization (the TUI).

### Phase 1: Instrumentation (The Application Side)

Your existing backend services do not need their code rewritten. By attaching the OpenTelemetry Java Agent to a Spring Boot JVM instance, OTel dynamically injects itself into the framework.

- When an HTTP request hits the API, the agent generates a `trace_id` and a root span.
- When the service makes a gRPC call, the agent injects the `trace_id` into the gRPC metadata headers.
- When a service produces a Kafka message, the agent wraps the `trace_id` into the Kafka record headers.
- The agent asynchronously pushes these generated span payloads (in JSON/Protobuf) to a specified endpoint.

### Phase 2: Ingestion Engine (The TUI Backend)

Inside the Python TUI application, a lightweight asynchronous HTTP server (using FastAPI or aiohttp) runs in a background thread.

- It listens on port `4318` (the standard OTLP HTTP port).
- As raw spans arrive over the network, it parses them and pushes them into an in-memory datastore (e.g., an SQLite database or a specialized Python dictionary).
- It continuously groups incoming spans by their shared `trace_id`.

### Phase 3: The State & UI Renderer (The TUI Frontend)

Using a modern terminal framework (like Python's Textual), the application draws the interface.

- **The Watcher:** The UI framework monitors the in-memory datastore for changes.
- **Tree Construction:** When a trace completes, it calculates the hierarchy using the `parent_span_id` attached to each span.
- **Interaction:** When the user navigates the tree with arrow keys, the UI retrieves the specific span's attributes (duration, errors, custom tags) and pretty-prints the JSON in the inspection panel.

## 6. Technical Stack Recommendations

To build this efficiently, the following stack is recommended:

- **TUI Framework:** Python with [Textual](https://github.com/Textualize/textual). Textual is asynchronous, uses a CSS-like styling engine, and has pre-built Tree and DataTable widgets that are perfect for this use case.

- **Local Ingestion:** Python with [FastAPI](https://fastapi.tiangolo.com/). It can easily spin up a background server to catch the HTTP POST requests containing the telemetry data.

- **Testing/Target Architecture:** Spring Boot (Java), gRPC, and Apache Kafka. (Testing the tool against a robust, multi-protocol backend ensures it handles complex edge cases like asynchronous message brokering).
